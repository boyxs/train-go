## 1. 目标与场景

让用户在小微书内建立并管理人与人的关系，为内容分发、私信、社区归属打底。

- **核心场景**：读到好内容 → 关注作者 → 在"我的关注/粉丝"管理关系 → 骚扰者拉黑屏蔽
- **痛点**：当前只有对内容的轻互动（点赞/收藏/评论），缺少对"人"的关注沉淀，作者无法积累粉丝、读者无法追踪创作者

## 2. 需求范围（已确认）

| 决策点 | 结论 |
|--------|------|
| 关系模型 | **单向关注边**（我关注的 / 关注我的粉丝 / 互相关注 mutual）+ **拉黑名单**；不做好友双向请求-接受 |
| 关注流 | **不含**（本期）。关系只做建立/解除/查询/计数；**关注流(feed) 后续单开独立服务**，relation 需预留干净的关系查询接口 + 关注/取关领域事件供未来 feed 服务消费，本期不嵌 feed 逻辑、不反向依赖 feed |
| 关联方式 | 用户对用户（`follower_id` → `followee_id`），与 `interaction`/`comment` 的 `biz+bizId` 平行的独立领域 |
| P0 功能 | 关注 / 取关 + 关注列表 / 粉丝列表（游标加载）+ 关注/粉丝计数 + 互关标记 + 拉黑 / 取消拉黑 + 黑名单 |
| 权限 | 计数与列表默认公开；关注/取关/拉黑需登录；拉黑级联解除双向关注 |
| 私信联动 | 关注非私信前置（可直接私信），但**被拉黑方无法私信**（拉黑作为 chat 的黑名单校验来源，接口留给 chat 调用）——P1 落地 |

## 3. 用户故事

| 优先级 | 角色 | 故事 |
|--------|------|------|

| P0 | 用户 | 查看「我的关注」列表，以便管理关注对象 |
| P0 | 用户 | 查看「我的粉丝」列表，未回关者可「回关」，以便建立互关 |

| P0 | 用户 | 拉黑某人（自动解除双向关注、对方无法再关注/私信我），以便屏蔽骚扰 |

| P2 | 用户 | 关注分组 / 备注名；「你可能认识的人」推荐关注 |

## 4. 核心流程 + 异常分支

```
关注：他人主页/作者卡 →「关注」
   ├─ [未登录] 引导登录
   ├─ [已拉黑对方] 提示「已拉黑，请先取消拉黑」
   ├─ [被对方拉黑] 提示「无法关注」（按钮置灰「无法关注」）
   ├─ [关注自己] 前端不展示关注按钮 / 后端拒绝
   └─ 成功：按钮→「已关注」（对方也关注我则「互相关注」），粉丝数 +1（乐观更新，失败回滚）

取关：「已关注 / 互相关注」→（可二次确认）→ 成功：按钮→「关注」，粉丝数 -1

拉黑：主页「⋯」→「拉黑」→ 二次确认弹窗（提示：解除双向关注 + 对方无法关注/私信/评论你）
   └─ 成功：双向关注解除、加入黑名单、关闭菜单

取消拉黑：黑名单页「取消拉黑」→ 从黑名单移除（**不自动恢复**先前的关注关系）

列表：profile「关注 / 粉丝」Tab → 游标加载更多 → 空态引导（「还没有关注任何人 / 还没有粉丝」）
```

## 6. API 接口

统一响应 `{ code, msg, data }`；认证 `x-access-token`。分页统一游标（对齐 `comment` 加载更多）。

| Method | 路径 | 说明 | 认证 |
|--------|------|------|------|
| POST | `/relation/follow` | 关注（body: `followeeId`） | JWT |
| POST | `/relation/unfollow` | 取关（body: `followeeId`） | JWT |
| POST | `/relation/followees` | 关注列表（`userId` + 游标）；含每人 `isMutual` | JWT 可选 |
| POST | `/relation/followers` | 粉丝列表（`userId` + 游标）；含每人 `isFollowedBack` | JWT 可选 |
| POST | `/relation/stat` | 关注数/粉丝数 + `isFollowing`/`isMutual`/`isBlocked`（对某人） | JWT 可选 |
| POST | `/relation/block` | 拉黑（body: `targetId`） | JWT |
| POST | `/relation/unblock` | 取消拉黑（body: `targetId`） | JWT |
| POST | `/relation/blocklist` | 黑名单列表（游标） | JWT |

> 用户资料（昵称/头像/简介）由 core 聚合 user 服务 `FindByIds` 填入 VO（对齐 comment 昵称聚合模式），relation 只存 id 关系。

## 7. 数据模型要点（细节留 architect 定案）

- `follow_relation`：`follower_id` + `followee_id` + `status`(软删) + `created_at:int64`；唯一索引 `(follower_id, followee_id)`；查询索引 `(followee_id)` 支撑粉丝列表。
- `user_block`：`uid` + `blocked_uid` + `created_at:int64`；唯一索引 `(uid, blocked_uid)`。
- 计数：`follower_cnt` / `followee_cnt` 走 Redis 计数器（对齐 `interaction` 的 `like_cnt`），DB 为准、缓存回填、写后维护、TTL 加 jitter。
- 互关判定：查双向 `follow_relation` 是否都存在（或计数/缓存标记）。

## 8. 边界与约束

| 约束 | 规则 |
|------|------|
| 自关注 | 禁止关注自己（后端拒绝，前端不展示按钮） |
| 幂等 | 重复关注/拉黑幂等，不重复计数 |
| 拉黑级联 | 拉黑时解除双向 `follow_relation`；被拉黑方 follow/私信被拒 |
| 取消拉黑 | 仅移出黑名单，不恢复关注 |
| 分页 | 游标分页，`pageSize ≤ 50`，关注时间倒序 |
| 时间 | 全链路 `int64` 毫秒（项目铁律） |
| 计数一致性 | 缓存计数与 DB 最终一致；异常降级为实时 count DB |
| 隐私 | 关系列表默认公开；隐私开关留 P1 |
| feed 解耦 | 关注流后续独立成服务；relation 对外暴露关系查询 API + 关注/取关领域事件（Kafka topic + JSON，两端各定义契约），不内嵌 feed、不反向依赖。粉丝列表查询需支撑未来 feed 写扩散（拉取某人全部粉丝 id） |

## 9. 验收标准

```
Given 已登录 A 未关注 B    When A 在 B 主页点「关注」   Then 按钮→「已关注」、B 粉丝数 +1、A 关注数 +1
Given A、B 互相关注         When 打开任一方主页/列表     Then 关注按钮显示「互相关注」并有「互相关注」标记
Given A 已关注 B            When A 点「已关注」取关       Then 按钮→「关注」、B 粉丝数 -1
Given A 拉黑 B              When 拉黑成功               Then A/B 双向关注解除、B 进入 A 黑名单、B 侧对 A 关注按钮置「无法关注」
Given B 在 A 的黑名单       When A 点「取消拉黑」         Then B 移出黑名单；A、B 关注关系不自动恢复
Given 用户无关注/粉丝       When 打开对应 Tab            Then 显示空态与引导文案
Given 关注列表 > pageSize   When 下滑点「加载更多」       Then 游标续拉下一页、不重不漏
```
