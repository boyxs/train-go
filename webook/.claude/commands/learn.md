# 项目学习

系统性地理解和学习指定项目/代码库，输出一份结构化的认知报告。

接受参数：项目路径或模块名。无参数时分析当前项目。

## Step 1: 项目全貌
- 读取 README / CLAUDE.md / go.mod / package.json 等入口文件
- 梳理目录结构和各目录职责
- 输出：项目一句话描述 + 技术栈概览

## Step 2: 架构分析
- 识别分层/模块划分方式
- 梳理模块间的调用关系和依赖方向
- 识别核心数据流（用户操作 → 前端 → 后端 → 数据库 → 响应）
- 识别共享资源（共享库、共享数据库、共享配置）

## Step 3: 核心业务流程
- 识别 3-5 个核心业务流程
- 对每个流程：追踪从入口到数据落库的完整链路
- 标注每个环节涉及的文件和模块

## Step 4: 代码模式识别
- 编码风格和命名规范
- API 层封装方式（路由注册、中间件、参数绑定）
- 错误处理模式（error wrapping / sentinel error / panic recovery）
- 数据库查询模式（ORM / raw SQL / 缓存策略）
- 依赖注入方式（Wire / 手动注入 / 全局变量）
- 测试模式（单元测试 / 集成测试 / mock 方式）

## Step 5: 关键文件索引
- 每个模块的入口文件
- 配置文件位置
- 路由定义位置
- 数据模型定义位置
- 工具函数和公共模块位置
- 依赖注入配置位置

## Step 6: 设计亮点与改进点
- 值得学习的设计模式和技巧
- 技术债和改进空间
- 性能瓶颈（N+1 查询、全量加载、缺失索引）
- 安全风险
- 缺失的测试覆盖

## 输出格式
```
# [项目名] 认知报告

## 项目概述
一句话 + 技术栈

## 架构
分层/模块关系 + 数据流

## 核心业务流程
流程名 → 完整链路（标注文件）

## 代码模式
API / 错误处理 / 数据库 / DI / 测试

## 关键文件索引
按模块列出入口和关键文件

## 亮点与借鉴
值得学习的设计

## 改进建议
技术债 / 性能 / 安全 / 测试
```
