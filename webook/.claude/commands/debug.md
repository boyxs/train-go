# 系统调试

> **铁律**: 没有根因分析，不尝试修复。
> 随机修复浪费时间并制造新 bug。系统化方法 15-30 分钟，随机尝试 2-3 小时。

## Phase 1: 问题定义
- 用一句话描述问题现象
- 预期行为 vs 实际行为
- 复现步骤（必须能稳定复现）
- 影响范围：哪些用户/场景受影响

## Phase 2: 信息收集
- 相关错误日志（zap 日志输出）
- 请求/响应数据
- 涉及的模块和文件（web / service / repository / dao / cache）
- 最近的代码改动（`git log --oneline -10` / `git diff`）
- 数据库状态（MySQL 数据是否符合预期）
- Redis 缓存状态（key 是否存在、TTL、值是否正确）

## Phase 3: 根因分析

按优先级排查，**每次只验证一个假设**：

1. **数据问题** — MySQL 数据是否符合预期，GORM 模型映射是否正确
2. **接口问题** — 请求参数绑定是否正确，JWT token 是否有效
3. **缓存问题** — Redis 缓存是否过期/脏数据，Cache-DB 一致性
4. **逻辑问题** — 条件判断、error 处理链是否正确
5. **并发问题** — goroutine 竞态、Redis 原子性、数据库事务隔离
6. **性能问题** — 慢查询、缺失索引、N+1 查询、连接池耗尽

### 根因追踪法
从错误现象**反向追踪**调用链：
```
错误表现 → web handler → service → repository → dao/cache → 根因
```
在每一层确认：输入是否正确？处理逻辑是否正确？输出是否正确？

## Phase 4: 修复方案
- 说明根因（一句话）
- 给出修复方案（如有多个，列出各自优缺点）
- **等确认方案后再改**

## Phase 5: 修复验证（不可跳过）
1. 写一个能复现原 bug 的测试用例
2. 确认测试**失败**（证明确实能捕获这个 bug）
3. 实施修复
4. 确认测试**通过**
5. 运行 `go build ./...` — 无编译错误
6. 运行 `go vet ./...` — 无静态分析问题
7. 运行相关测试 `go test ./internal/...` — 无回归

## Phase 6: 防止复发
- 刚才写的测试用例保留（防止回归）
- 记录到 `memory/`（feedback 类型）
- 如果是常见陷阱，考虑在 CLAUDE.md 中补充

## 红旗自检
如果你正在想以下任何一条，**立即停下**：
- "先试试改这里看看" → 没有假设的修复 = 随机尝试
- "大概是这个问题" → "大概"不是根因，继续查
- "快速修一下" → 快速修 = 慢速调试（之后的）
- "已经试了 2 个修复了，再试一个" → 如果 2 个都没用，说明根因判断错了，回到 Phase 3
- "先改了再说，不行再回退" → 这就是随机修复的定义

## 借口表
| 借口 | 现实 |
|------|------|
| "我知道是什么问题" | 那就先证明：写测试复现它 |
| "改动很小，不会有副作用" | 小改动引入的回归最难发现 |
| "来不及写测试了" | 不写测试 = 不知道修没修好 |
| "日志看起来正常" | "看起来"不是证据，跑测试才是 |
