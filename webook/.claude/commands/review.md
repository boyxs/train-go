# 代码审查

对当前改动做一次完整的代码审查。

> **原则**: 技术严谨，不做社交表演。
> 禁止回应："没什么问题！"、"代码写得很好！" — 这不是审查，是敷衍。

## 审查方式

先运行 `git diff` 查看所有改动，然后逐项检查。

## Stage 1: 规格合规性

改动是否匹配需求？
- [ ] 该实现的都实现了（没有遗漏）
- [ ] 没有过度构建（没有需求之外的功能）
- [ ] 分层正确：Handler → Service → Repository → DAO/Cache，无跨层调用

## Stage 2: 代码质量

### 2.1 错误处理
- [ ] error 全部处理，无忽略的 error
- [ ] error wrapping 正确（`fmt.Errorf("xxx: %w", err)`）
- [ ] 数据库连接失败、Redis 超时、外部 API 异常是否处理

### 2.2 安全性
- [ ] 无 SQL 注入风险（用户输入通过 GORM 参数化查询）
- [ ] 无未鉴权接口（需要认证的路由在 JWT 中间件之后）
- [ ] 无敏感数据泄露（密码哈希、token 不返回给前端）

### 2.3 性能
- [ ] 无 N+1 查询（循环内的数据库查询）
- [ ] 无全量加载（大表查询有分页或条件过滤）
- [ ] 缓存策略合理（热点数据有 Redis 缓存，缓存失效有处理）
- [ ] 数据库字段有合适的索引

### 2.4 并发安全
- [ ] Redis 操作需要原子性的地方用了 Lua 脚本
- [ ] 共享状态有适当的同步机制
- [ ] 数据库事务隔离级别合适

### 2.5 项目规范
- [ ] 方法命名语义明确（`FindXx` / `PageXxs` / `ListXxs` / `CreateXx` / `UpdateXx` / `DeleteXx`）
- [ ] receiver 命名用类型首字母小写（`func (s *Service) Run()`）
- [ ] 无 `fmt.Println`，用 `zap` 日志
- [ ] 无硬编码（URL、端口、阈值应为配置或常量）
- [ ] 测试假数据有 `TODO` 注释块包裹

### 2.6 可读性
- [ ] 命名清晰，无歧义
- [ ] 超过 50 行的函数是否需要拆分
- [ ] 复杂逻辑有必要的注释

## 问题分级

对发现的每个问题标注严重程度：

- **Critical** — 必须修复才能提交（安全漏洞、数据丢失、编译错误）
- **Important** — 应该修复再提交（错误处理缺失、性能问题、回归风险）
- **Suggestion** — 建议优化但不阻塞（命名改进、代码简化、注释补充）

## 输出格式
```
## 审查结果

### Critical
- [文件:行号] 问题描述 + 修复建议

### Important
- [文件:行号] 问题描述 + 修复建议

### Suggestion
- [文件:行号] 问题描述 + 改进建议

### 总结
- **结论**: 通过 / 有问题需修复 / 建议优化
- **Critical**: N 项
- **Important**: N 项
- **Suggestion**: N 项
```

## 下一步
- Critical/Important 问题修复后 → 重新 `/review`
- 审查通过后 → `/done` 做最终检查
