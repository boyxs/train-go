# TDD 工作流

> **铁律**: 没有失败的测试，不写生产代码。
> 没有例外。不管多简单。不管多赶时间。

按照严格的 Red → Green → Refactor 循环来实现当前需求。

## Checklist

### 1. 理解需求
- 明确要实现什么功能
- 确认输入、输出、边界条件
- 列出所有需要覆盖的测试用例：
  - 正常路径（happy path）
  - 空数据 / nil / 零值
  - 无效输入 / 参数校验失败
  - 权限不足 / JWT token 无效或过期
  - MySQL 连接失败 / Redis 超时 / 外部 API 异常
  - 并发场景（FindOrCreate 竞态、Redis 原子操作）
- **等确认用例列表后再继续**

### 2. RED — 先写一个失败的测试
- 只写**一个**最小的测试用例（不是全部）
- 测试必须：
  - 命名清晰：`TestXxxService_Method_Scenario`
  - 调用真实代码（不是 mock 一切）
  - 断言一个具体行为
- 运行测试：`go test ./internal/... -run TestXxx -v`
- **确认测试失败**（因为还没写实现）
- **确认失败原因是预期的**（编译错误或断言失败，不是配置问题）

### 3. GREEN — 写最小实现
- 只写刚好让这个测试通过的代码
- **不要**：
  - 提前写下一个测试的实现
  - 做"顺便"的优化
  - 加"以后可能用到"的功能
- 运行测试：确认**通过**
- 运行全量测试：`go test ./...` 确认无回归

### 4. REFACTOR — 重构
- 测试全绿后，重构实现代码：
  - 提取重复逻辑
  - 优化命名
  - 拆分过长的函数
- **每次重构后运行测试**，确认仍然全部通过
- 重构只改内部实现，不改外部行为

### 5. 循环
- 回到 Step 2，写下一个测试用例
- 重复 RED → GREEN → REFACTOR 直到所有用例覆盖

### 6. 完成
- 运行 `go test ./... -v` 贴出完整输出
- 报告：总用例数、通过数、覆盖的场景

## 测试规范

- 单元测试：`testing` + `testify/assert` + `go.uber.org/mock`（gomock）
- 集成测试：放 `internal/integration/`，用 `testify/suite`，需要真实 MySQL + Redis
- Mock 生成：接口变更后运行 `make -f win.mk mockgen`
- Wire 生成：集成测试的 wire 在 `internal/integration/setup/`
- 表驱动测试（table-driven tests）用于多场景覆盖
- 每个测试独立，不依赖执行顺序
- 测试假数据用 `// ===== TODO: 测试假数据，正式环境删除 START =====` 包裹

## Red Flags

如果你正在想以下任何一条，**立即停下**：

| 借口 | 现实 |
|------|------|
| "先写实现，再补测试" | 后写的测试什么都证明不了 — 它从未失败过 |
| "太简单了，不需要测试" | 简单的代码也会出错，而且"简单"是你现在的判断 |
| "这个没法测试" | 几乎总是因为设计有问题，先改设计 |
| "测试会拖慢进度" | 不写测试 = 不知道代码对不对 = 更慢 |
| "先写完再重构" | 没有测试的重构是在碰运气 |
| "Mock 太多了，测试没意义" | 减少 mock，测试更有意义的层（集成测试） |
